## 4.7 Architecture des API REST

### 4.7.1 Présentation

L'application adopte une architecture **REST (Representational State Transfer)** pour assurer la communication entre le frontend développé avec Flutter et le backend développé avec Django REST Framework.

Les API REST exposent les différentes fonctionnalités de l'application sous forme de services accessibles via le protocole HTTP. Les échanges de données sont réalisés au format **JSON**, garantissant une communication rapide, légère et indépendante de la plateforme cliente.

Cette architecture permet de séparer la couche de présentation de la logique métier, facilitant ainsi la maintenance, les évolutions futures et l'intégration avec d'autres applications ou services.

### 4.7.2 Principe de fonctionnement

Le fonctionnement des API REST repose sur le modèle client-serveur :

1. Le frontend Flutter envoie une requête HTTP au backend.
2. Le backend reçoit la requête et vérifie les droits d'accès.
3. La logique métier est exécutée.
4. Les données sont enregistrées ou récupérées dans PostgreSQL.
5. Le backend renvoie une réponse JSON au frontend.

### 4.7.3 Architecture des échanges

Les échanges entre les différents composants du système sont réalisés selon le schéma suivant :

Flutter → API REST → Django REST Framework → PostgreSQL

Les réponses retournées par le backend sont normalisées afin de garantir une communication homogène entre les différents modules.

### 4.7.4 Authentification

Les API sont sécurisées à l'aide d'un mécanisme d'authentification basé sur des jetons JWT (JSON Web Token).

Après une authentification réussie, le serveur délivre un jeton d'accès qui devra être présenté dans l'en-tête des requêtes nécessitant une autorisation.

### 4.7.5 Organisation des API

Les services REST sont organisés par domaine fonctionnel.

Les principales catégories d'API sont :

* API d'authentification
* API des utilisateurs
* API des clients
* API des services
* API des commandes
* API des paiements
* API de gestion de la caisse
* API du tableau de bord
* API des rapports
* API des notifications

Cette organisation facilite la maintenance et la compréhension du système.

### 4.7.6 Méthodes HTTP

Les API utilisent les principales méthodes HTTP :

* **GET** : consulter des informations.
* **POST** : créer une nouvelle ressource.
* **PUT** : modifier entièrement une ressource.
* **PATCH** : modifier partiellement une ressource.
* **DELETE** : supprimer une ressource.

### 4.7.7 Format des données

Toutes les requêtes et réponses utilisent le format JSON.

Chaque réponse contient un code HTTP approprié ainsi que les données demandées ou un message décrivant le résultat de l'opération.

### 4.7.8 Sécurité

La sécurité des API repose sur plusieurs mécanismes :

* authentification JWT ;
* contrôle des rôles et des permissions ;
* validation des données reçues ;
* gestion centralisée des exceptions ;
* limitation des accès aux ressources sensibles.

### 4.7.9 Évolutivité

L'architecture REST retenue permet d'ajouter facilement de nouveaux endpoints ou de nouvelles fonctionnalités sans modifier les interfaces existantes. Cette modularité garantit une évolution progressive de l'application tout en assurant la compatibilité entre le frontend et le backend.
